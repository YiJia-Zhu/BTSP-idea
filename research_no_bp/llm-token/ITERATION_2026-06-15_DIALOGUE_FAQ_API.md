# Iteration 2026-06-15: Dialogue-Style FAQ API Prototype

## Purpose

Move the FAQ prototype closer to an online interaction setting. Instead of learning only canonical policy notes, the memory is updated from dialogue-style instructions and evaluated on paraphrased questions.

## Code

- `online_memory_faq_api_experiment.py`
- Added `--train-style dialogue`.
- Added `--eval-style paraphrase`.
- Added generated paraphrase questions and broader intent aliases.
- Updated access-policy scoring to accept equivalent answers such as `badge required`.

## Commands

```bash
python -m py_compile online_memory_faq_api_experiment.py

python online_memory_faq_api_experiment.py \
  --out-dir output/online_memory_faq_dialogue_dry \
  --dataset generated --train-style dialogue --eval-style paraphrase \
  --fact-limit 64 --api-limit 16

python online_memory_faq_api_experiment.py \
  --out-dir output/online_memory_faq_dialogue_256_dry \
  --dataset generated --train-style dialogue --eval-style paraphrase \
  --fact-limit 256 --api-limit 16

API_KEY=... python online_memory_faq_api_experiment.py \
  --out-dir output/online_memory_faq_dialogue_api_run \
  --dataset generated --train-style dialogue --eval-style paraphrase \
  --fact-limit 64 --api-limit 16 --run-api --api-timeout 90
```

## Interpretation

This strengthens the API-compatible prototype: the model can answer paraphrased FAQ questions after the no-BP memory is updated from dialogue-style instructions, without raw training-example replay.

The result remains bounded:

- The memory still has a fact schema and stores canonical answer values.
- The paraphrases are generated by deterministic templates.
- This is not yet an open-ended multi-turn support conversation or GPT-level continual learner.

## Next Steps

1. Add overwrite and deletion tests for natural FAQ values.
2. Add multi-turn sessions where facts are introduced, revised, and queried later.
3. Replace the hand-authored alias router with learned semantic keys or API embeddings while retaining no raw examples.
